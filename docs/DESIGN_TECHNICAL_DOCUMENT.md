## 1. 比赛准备和调研

### 1.1 赛题理解与评审场景

本项目对应“proj18 面向小型操作系统的分析比对智能体系统设计”。赛题的核心不是重新实现一个操作系统，而是构建一个能够读取小型操作系统仓库、抽取关键技术特征、与历史作品和参考内核进行对比、并生成可解释报告的智能体系统。

结合初赛评审方式，系统需要同时服务两类场景：第一，评委查看开源仓库、文档和演示视频时，能够快速理解项目解决的问题、技术路线和可信边界；第二，使用者输入新的比赛仓库后，系统能够给出结构化、可复核的分析结果，而不是只给出泛泛的自然语言评价。

因此，本项目从一开始就把“证据链”作为基本要求：报告中的结论必须尽量绑定到文件路径、符号、构建入口、README 或文档片段，避免把 LLM 的语言生成能力误用为无依据裁定。

### 1.2 历史样本与开源资料调研

项目调研了往届操作系统比赛作品、教学内核、架构参考项目和赛事组提供的 collected-data 样本。现有历史基线库以本地可复现为原则，只把能够成功 clone 或已经验证可用的仓库纳入正式样本清单。

当前样本库共包含 141 个历史样本，其中包括已验证代表作品、教学基线、架构参考项目，以及从赛事组 collected-data 中成功拉取的 120 个历史作品。赛事组 collected-data 原始记录为 168 条，其中 48 条因为仓库不可访问、认证限制、地址失效或网络拉取失败等原因未纳入正式本地基线库，并在导入说明中保留失败记录，避免在演示和评审中混淆样本来源。

样本库不是把所有历史仓库合并为一个“大仓库”再做简单文本查重，而是为每个仓库分别构建画像，再在输入作品分析时按架构、语言、规模、OS 机制覆盖度和功能相似度选择 Top N 历史样本进行对比。

### 1.3 技术路线选择

系统采用“静态仓库分析 + OS 语义画像 + 历史样本检索 + 报告生成 + 可选 LLM 增强”的路线。静态分析负责提供确定性证据，LLM 只在证据受控的上下文中承担摘要、对比表达和报告组织工作。

技术上优先使用 Python 实现命令行工具、数据处理和报告生成，降低部署门槛；网页部分采用静态站点方式，便于在 ECS/Nginx 上发布为公开链接。这样的路线适合初赛阶段的开源仓库审查和视频演示，也便于后续扩展到批量输入多个比赛仓库。

## 2. 系统框架和模块设计

### 2.1 总体架构

KernelSage 的整体流程可以概括为：输入仓库路径或 Git 地址，系统先完成文件采集和语言识别，再解析代码符号和构建线索，然后生成 KernelProfile 画像，随后从历史样本库中选择相近基线，最后输出 Markdown 报告、网站页面或审计结果。

核心模块位于 `src/os_agent/`，命令入口位于 `scripts/`。数据侧主要包含 `data/samples/manifest.json`、历史样本目录、本地画像缓存和报告输出目录。网站侧主要包含 `site/` 下的静态页面和构建脚本。

系统分层如下：

- 输入层：接收本地仓库、历史样本清单、配置文件和可选 LLM 参数。
- 分析层：完成仓库采集、符号解析、OS 机制识别和画像生成。
- 对比层：从历史样本库中选择相近项目，并生成差异、相似点和代码级线索。
- 生成层：输出 Markdown 报告、评分摘要、LLM 增强报告和网页数据。
- 审计层：检查报告引用、样本清单、LLM 输出和网站内容是否自洽。

### 2.2 数据采集与仓库画像模块

`collector.py` 负责扫描仓库文件、README、文档、构建脚本和 manifest 元数据，形成后续分析所需的基础输入。采集时会过滤常见低价值目录，避免把构建产物、依赖目录或隐藏缓存误当作核心实现。

`parser.py` 负责从 Rust、C、C++ 和汇编文件中提取函数、类型、宏、模块声明和符号线索。由于小型 OS 项目经常混合使用 Rust/C/ASM，也可能包含 `.cc`、`.cpp`、`.hpp` 等 C++ 文件，解析器需要覆盖多语言入口。

`analyzer.py` 将采集结果转换成 KernelProfile。画像围绕小型操作系统常见机制组织，包括启动与引导、内存管理、任务调度、系统调用、文件系统、驱动与中断、网络与进程管理等维度。画像结果不是只统计代码行数，而是尽量提取“这个仓库实现了哪些 OS 机制、证据在哪里、成熟度如何”。

### 2.3 历史样本选择与对比模块

`selector.py` 根据输入仓库画像和历史样本画像选择相近基线。选择时综合考虑架构、语言、规模、仓库类型、OS 机制重合度和历史样本来源，避免只因为某个关键词相同就把两个项目判定为强相关。

`agent.py` 和相关对比逻辑负责生成历史样本比较结果。比较内容包括总体相似度、机制覆盖重合、输入作品相对优势、风险点、可补充证据，以及代码级相似线索。代码级线索包括路径相似、函数或符号相似、类型或结构体相似、宏和常量相似、局部片段 token/结构相似等。

该模块的目标是为评审提供“可追问”的参考，而不是给出简单查重结论。报告会把功能相似和代码相似区分开，避免把操作系统领域天然相似的模块名称直接解释为抄袭。

### 2.4 LLM 增强与报告审计模块

`llm.py` 提供可选 LLM 调用能力，`llm_audit.py` 用于审计 LLM 报告。系统默认可以不调用 LLM，使用规则化分析生成基础报告；当启用 LLM 时，提示词会限制模型只能基于已采集证据、历史样本画像和明确引用进行表达。

LLM 增强主要用于提升报告的可读性和对比组织能力，而不替代静态分析。为了控制幻觉风险，系统提供报告审计能力，检查报告是否存在未知引用、缺失关键章节、过度断言或没有证据支撑的判断。

### 2.5 网站展示与演示模块

网站部分位于 `site/`，通过静态页面展示项目介绍、历史基线库、案例报告和操作入口说明。`scripts/build_site.py` 用于根据当前样本库和报告数据生成网站内容，`scripts/serve_site.py` 用于本地预览。

由于比赛要求提供可访问的网站链接，项目已经将静态站点部署到 ECS，并通过 Nginx 暴露公网访问。这样评委不需要连接本机 `localhost`，可以直接访问公开页面查看基线库、项目说明和演示材料。

## 3. 开发计划

### 3.1 阶段一：仓库扫描和基础报告

第一阶段目标是让系统能够读取一个输入仓库，并输出基础结构报告。该阶段重点完成仓库遍历、文件类型识别、README/文档采集、构建入口识别和基础统计。

验收标准是：对一个本地 OS 仓库运行命令后，能够生成包含语言分布、目录结构、核心文件、构建线索和初步机制判断的 Markdown 报告。

### 3.2 阶段二：OS 语义画像与历史样本库

第二阶段目标是把普通代码统计提升为 OS 语义画像。系统需要识别启动、内存、调度、系统调用、文件系统、驱动、中断、网络等机制，并保存为结构化 KernelProfile。

同时建立历史样本 manifest，区分教学基线、架构参考、已验证代表作品和赛事历史作品。每个样本独立建档，后续输入新作品时通过画像检索相近历史项目。

### 3.3 阶段三：对比报告、代码相似线索和 LLM 审计

第三阶段目标是生成更接近评审需要的对比报告。报告除了说明输入作品实现了什么，还要回答“它和哪些历史作品相近、相近在哪里、差异在哪里、有哪些证据”。

此阶段加入代码级相似线索，覆盖路径、符号、结构体、宏和局部片段；同时加入 LLM 增强和 LLM 审计，确保自然语言报告不脱离证据。

### 3.4 阶段四：样本扩展、网站发布和文档收敛

第四阶段围绕初赛交付展开：导入赛事组 collected-data，扩展历史基线库；为全部历史样本预构建画像，减少演示时重复分析；生成公开网站页面；整理 README、操作手册、报告审计说明、样本导入说明和本设计技术文档。

该阶段的验收标准是：仓库可以被评委直接检查，网站可以公网访问，操作手册可以支撑视频演示，报告样例可以说明系统输出的详细程度和可信边界。

## 4. 比赛过程中的重要进展

### 4.1 从单仓库分析扩展为历史作品比对

项目早期重点是对单个输入仓库生成描述评价，后来扩展为“输入作品 vs 历史样本库”的比较模式。这样报告不只说明当前仓库内部情况，还能把它放到往届作品、教学基线和架构参考项目中进行横向定位。

以 `oskernel2024-aabcb` 这类输入作品为例，系统可以先为输入仓库构建画像，再与历史样本库中相近项目进行对比，输出相似点、差异点、代码级线索和改进建议。

### 4.2 从 Top 3 对比扩展到 Top 5 对比

为了让比较结果更充分，系统将历史样本对比数量从 Top 3 扩展到 Top 5。这样可以减少偶然样本对结论的影响，让报告覆盖更多相近实现路径。

同时系统明确了 API 消耗边界：如果只使用本地静态分析和已经构建好的画像缓存，Top 5 不会增加 LLM API 消耗；只有在显式启用 LLM，并要求 LLM 逐项分析更多历史样本时，API token 消耗才会增加。

### 4.3 历史基线库扩展到 141 个样本

项目将赛事组提供的 collected-data 纳入历史基线建设。原始 collected-data 包含 168 条记录，成功拉取并整理进样本库的为 120 条，结合此前已验证样本后，正式历史基线库达到 141 个本地可用样本。

这些样本统一归类为“赛事历史作品”等可解释来源，不再把赛事组 collected-data 单独标成难以理解的内部类别。系统还为全部 141 个样本构建了画像缓存，后续输入新作品时可以直接复用历史画像。

### 4.4 建立可公开访问的网站

本地 `http://localhost:8080/` 只能在开发机器访问，无法满足评委远程查看要求。因此项目将静态网站部署到阿里云 ECS，通过公网 IP 提供访问。

当前网站包含项目首页、历史基线库页面和相关展示内容，基线库页面能够反映最新的 141 个历史样本。该部署方式降低了评委访问门槛，也使演示视频中的链接与实际提交材料保持一致。

## 5. 遇到的主要问题和解决方法

### 5.1 历史仓库来源复杂，部分仓库无法 clone

痛点：赛事组 collected-data 中的仓库地址来源复杂，存在仓库删除、权限限制、地址变更、网络访问失败等情况。如果把所有原始记录都直接写入样本库，会造成后续演示时无法复现，也会影响评委对基线库可信度的判断。

解决方法：项目只把成功 clone 并能在本地检查的仓库纳入正式样本 manifest。对 168 条原始记录逐条尝试拉取，最终纳入 120 条可用样本，失败的 48 条保留在导入说明中，作为不可用记录而不是伪装成已验证样本。这样既利用了赛事组资料，又保持了本地基线库的可复现性。

### 5.2 样本来源分级不清，容易造成获奖信息误用

痛点：历史样本可能来自获奖项目、普通赛事项目、教学内核或架构参考。如果在报告中不区分来源，容易把普通样本误写成获奖样本，或者把教学项目当作赛事作品进行评价。

解决方法：manifest 中维护样本类别、来源层级、奖项信息和来源链接。只有经过验证的项目才写入明确奖项信息；赛事组 collected-data 中后续补充的样本统一归入赛事历史作品，不使用未经确认的获奖表述。报告生成时按样本元数据展示来源，减少过度宣传和事实错误。

### 5.3 关键字匹配容易误判 OS 机制

痛点：早期规则容易因为文件名、宏名或普通注释命中关键词而误判。例如 `sys` 可能只是普通变量或路径片段，不一定代表系统调用；宏或配置项也可能只是占位，不代表完整机制实现。

解决方法：分析器改为结合路径、符号、文件类型、上下文和低价值目录过滤进行判断。对宏注释、构建产物、第三方目录和弱关键词降低权重；对启动入口、trap/interrupt 处理、页表、调度器、系统调用分发表等更强证据提高权重。这样报告能更接近真实 OS 实现，而不是关键词列表。

### 5.4 C++、unikernel、RTOS 等非典型内核覆盖不足

痛点：小型 OS 项目不一定都是 Rust 或 C 教学内核，也可能是 C++ 内核、unikernel、RTOS 或应用内核混合形态。若只支持少数扩展名和传统模块命名，容易漏掉真实实现。

解决方法：解析器扩展支持 `.cc`、`.cpp`、`.cxx`、`.hh`、`.hpp`、`.hxx` 等 C++ 文件，并在机制判断上允许不同项目形态有不同证据组合。系统不要求每个仓库都必须具备完整宏内核结构，而是根据项目定位解释其启动、内存、任务、驱动或运行时机制。

### 5.5 功能重合和代码相似边界不清

痛点：操作系统作品天然会出现相似目录名和模块名，如 `memory`、`trap`、`syscall`、`fs`、`scheduler`。如果只看功能重合，很容易把正常领域共性误解为代码重复；如果完全不看代码线索，又无法支撑查重场景。

解决方法：系统把“功能相似”和“代码级相似线索”拆开表达。功能相似用于说明两个项目解决的问题相近；代码级线索则进一步检查路径、函数名、类型名、宏、常量和局部片段结构。报告会明确这些线索是辅助判断依据，不直接等同于抄袭裁定，最终仍需要结合仓库提交历史、说明文档和人工复核。

### 5.6 LLM 输出存在幻觉和越界判断风险

痛点：LLM 擅长组织语言，但如果不受约束，可能编造文件、误引历史样本、给出超出证据范围的判断，甚至把系统不能证明的内容写成确定结论。

解决方法：项目把 LLM 设计为可选增强层，不作为唯一分析来源。提示词要求模型基于已有证据表达；报告生成后使用审计工具检查未知引用、缺失章节和过度断言。没有启用 LLM 时，系统仍可生成规则化报告；启用 LLM 时，也保留缓存、dry-run 和审计流程，便于控制 API 成本和输出质量。

### 5.7 大样本库重复分析耗时

痛点：历史基线库扩展到 141 个样本后，如果每次输入比赛作品都重新扫描全部历史仓库，会显著增加演示时间，也会影响批量分析 200 份比赛仓库时的效率。

解决方法：系统为历史样本建立 ProfileCache。已经构建过的历史样本画像会缓存到本地，后续分析输入作品时只需要读取缓存并计算相似度。当前 141 个历史样本画像已全部构建完成，后续输入新仓库时不会重复读取每个历史仓库的全部代码，除非样本文件发生变化或缓存被清理。

### 5.8 本地演示网站无法被评委直接访问

痛点：开发阶段网站运行在 `localhost:8080`，只能本机访问。初赛要求提交可访问的网站链接，评委无法通过本地地址查看内容。

解决方法：项目将静态站点部署到阿里云 ECS，并使用 Nginx 提供公网访问。网站内容来自仓库中的 `site/` 目录和构建脚本，包含项目说明、历史基线库和演示材料。这样评委点击公开链接即可查看页面，不需要本地运行项目。

## 6. 作品特征描述

### 6.1 面向小型操作系统的证据链分析

KernelSage 的核心特征是把小型 OS 仓库转换成可解释画像。系统不仅统计代码规模，还会识别启动路径、内存管理、任务调度、系统调用、文件系统、驱动、中断、网络等机制，并尽量给出对应文件和符号证据。

这种证据链设计适合操作系统比赛评审：评委可以沿着报告中的路径回到仓库源码，检查结论是否成立；参赛者也可以据此发现文档缺口、实现短板和演示重点。

### 6.2 历史作品画像和基线对比

系统维护独立的历史样本 manifest 和画像缓存。每个历史样本都有自己的来源、类别和画像，不会把所有仓库混成一个不可解释的数据池。

当输入新作品时，系统根据画像相似度选择 Top 5 历史样本，输出相似点、差异点和代码级线索。这样报告能说明“和哪些历史作品相近”，也能说明“相近的依据是什么”。

### 6.3 可控 LLM 增强

LLM 在本项目中承担辅助写作和对比表达工作。系统通过证据输入、提示词约束、报告审计和缓存机制控制 LLM 风险。

这种设计避免了两类极端：一类是完全不用 LLM，导致报告表达生硬、人工整理成本高；另一类是完全依赖 LLM，导致报告不可复核。KernelSage 选择让静态分析提供事实基础，让 LLM 在边界内提升可读性。

### 6.4 可复现的命令行和网页展示

项目同时提供命令行和网页两种使用方式。命令行适合批量分析、生成报告和运行审计；网页适合评委快速浏览项目能力、历史样本库和演示案例。

静态网站可以本地预览，也可以部署到服务器公网访问。由于网站由仓库数据生成，基线库数量、样本来源和报告展示能与代码仓库保持一致。

团队名称为“一定要以人类的身份赢啊”，成员为鲍灿辉、石雅禛，学校为天津师范大学。项目围绕面向小型操作系统的分析比对智能体展开，目标是在初赛阶段提交可运行代码、可访问网站、演示视频和配套文档。

鲍灿辉主要负责整体技术路线、核心分析流程、历史样本导入、报告生成、网站部署和演示流程整理。具体包括仓库画像、历史样本对比、LLM 调用边界、ECS 部署、操作手册和提交材料检查。

石雅禛主要参与资料整理、文档组织、测试样例检查、报告模板校对和演示材料辅助准备。具体包括根据初赛要求核对章节结构、整理比赛过程材料、协助检查报告可读性和操作流程完整性。

## 9. 比赛收获

### 9.1 对操作系统作品分析的理解

通过项目建设，我们更清楚地认识到，小型操作系统作品不能只用代码量或关键词评价。启动流程、内存管理、调度、系统调用、文件系统、驱动和中断等机制之间存在关联，分析报告必须尽量说明证据来源和实现边界。

同时，不同作品可能选择不同路线：有的偏教学内核，有的偏架构实验，有的偏驱动或运行时，有的偏 unikernel 或 RTOS。合理评价需要尊重项目定位，而不是套用单一模板。

### 9.2 对智能体工程边界的理解

本项目让我们认识到，LLM 适合帮助组织复杂信息，但不适合在没有证据的情况下直接做技术裁定。面向评审和查重辅助场景时，智能体必须有明确的输入、缓存、引用、审计和失败处理机制。

因此，KernelSage 把确定性分析放在前面，把 LLM 放在可控增强层，并通过报告审计减少幻觉。这种工程边界比单纯追求自然语言流畅更重要。

### 9.3 对工程交付的理解

比赛提交不是只提交代码，还需要 README、设计文档、操作手册、网站链接、报告样例和演示视频形成闭环。项目在后期补齐了公开网站、历史基线库说明、样本导入记录和技术设计文档，使评委能够从多个入口理解作品。

这次开发也强化了我们对可复现性的理解：样本来源要可解释，报告结论要可检查，网站内容要能被外部访问，命令和文档要能支撑演示。只有这些环节同时成立，系统才真正具备比赛交付价值。
