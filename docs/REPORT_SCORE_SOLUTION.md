# 描述报告摘要评分问题与解决方案

## 问题

描述报告已经能按 OS 维度展示结论、源码证据、代码片段和复核建议，但如果缺少一个总览性的摘要评分，导师或评审仍然需要逐段阅读才能判断仓库整体成熟度。

直接加入评分也有风险：

- 如果分数由 LLM 主观生成，容易出现不可复核、不稳定和 token 成本增加的问题。
- 如果分数只看关键词数量，可能把弱证据、注释命中或误命中误算成高分。
- 如果分数不说明依据，容易被误解为比赛官方评分或最终质量裁定。
- 如果未确认维度不影响分数，报告会掩盖系统调用、文件系统、驱动等关键机制缺失。

因此，摘要评分必须满足三个约束：可解释、可复核、不额外消耗 LLM API。

## 解决方案

新增本地规则版“摘要评分/成熟度等级”，写入描述报告的 `摘要评分` 小节。

评分由已有静态分析结果派生，不调用 LLM：

| 评分项 | 分值 | 依据 |
| --- | --- | --- |
| OS 机制覆盖 | 80 | 调度、内存、系统调用、文件系统、同步、中断、驱动 7 个维度的确认情况 |
| 构建入口 | 10 | 是否识别到 Makefile、Cargo.toml、CMakeLists.txt 等构建入口 |
| 证据健康度 | 10 | self-check 中关键结论证据覆盖率和无效证据引用数 |

OS 维度采用加权评分：

| 维度 | 权重 |
| --- | --- |
| 调度与任务管理 | 14 |
| 内存管理 | 16 |
| 系统调用 | 12 |
| 文件系统 | 10 |
| 同步机制 | 10 |
| 中断与异常 | 10 |
| 设备驱动 | 8 |

高置信维度获得完整权重，中置信维度获得 70% 权重，未确认维度不得分。这样可以避免把“未确认”包装成成熟实现。

## 成熟度等级

| 分数 | 等级 | 含义 |
| --- | --- | --- |
| 85-100 | A 级：机制完整、证据充分 | 主要 OS 机制覆盖较完整，证据链较充分 |
| 70-84 | B 级：主要机制较完整 | 核心机制较多，但仍有部分维度缺失或未确认 |
| 50-69 | C 级：原型可用但覆盖不足 | 能看到部分 OS 实现线索，但完整度不足 |
| 0-49 | D 级：信息不足或实现线索较少 | 缺少足够源码证据支撑整体评价 |

## 当前验证结果

已用两个代表样本验证评分行为：

| 样本 | 评分结果 | 说明 |
| --- | --- | --- |
| `xv6-public` | A 级，100/100 | 7 个 OS 维度全部确认，构建入口确认，证据覆盖率 100% |
| `freertos-kernel` | B 级，70/100 | 调度、内存、同步、中断确认；系统调用、文件系统、设备驱动未确认 |

验证命令：

```powershell
$env:PYTHONPATH='src'; python -m unittest discover -s tests
python -m compileall src scripts\kernelsage.py
python scripts\kernelsage.py describe data\samples\xv6-public --repo-id xv6-public --rebuild-profile-cache
python scripts\kernelsage.py describe data\samples\freertos-kernel --repo-id freertos-kernel --rebuild-profile-cache
```

## 边界说明

- 该评分不是比赛官方评分，也不是最终质量裁定。
- 该评分只反映当前规则和当前证据库下的可解释成熟度。
- 未确认维度会拉低分数，但不等于仓库一定没有相关实现，仍需要人工复核。
- 评分默认由本地静态分析生成，不调用 DeepSeek API，不增加 token 成本。

## 答辩表述

我们没有让 LLM 主观给项目打分，而是把摘要评分建立在已确认维度、源码证据和 self-check 结果上。评分只是帮助评审快速理解仓库成熟度的入口，具体结论仍然可以回溯到文件、行号和代码片段。这样既提高了报告可读性，也避免了黑盒评分和 API 成本问题。
